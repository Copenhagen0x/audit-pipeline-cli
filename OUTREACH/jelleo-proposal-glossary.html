Reading Companion · Glossary

Every technical term in the funding proposal, in proposal order

A reference dictionary you can read alongside the funding proposal. Terms appear here in the order they appear in the proposal (Page 1 → Page 25), grouped by section. Each entry is a one-to-three-sentence plain-English explanation. Read this side-by-side with the proposal to understand every word, especially the technical Solana / DeFi / cryptography / formal-verification / finance terms.

How to use: Open the proposal PDF in one window and this glossary in another. As you encounter a term you don't know in the proposal, scan to the matching section here. Sections are numbered the same as the proposal (§01, §02, etc.).

Conventions: Words in monospace are code/CLI commands or technical identifiers. Words in italics are emphasis. Bold terms in the entry-name column are the lookup keyword.

Cover Pages 1–2 — Title page 10 terms

Jelleo
The brand/company name. Pronounced "jell-EE-oh". Lowercase wordmark on the website, uppercase in formal headers.
Autonomous Immune System
Metaphor for the product: software that detects threats, propagates antibodies (fix patterns), and adapts — without human intervention. The architectural identity of Jelleo.
Solana DeFi
Decentralized Finance protocols built on the Solana blockchain — perpetual exchanges, lending, AMMs, etc. "DeFi" = financial services running on smart contracts instead of through a bank.
Underwriting layer
The market positioning of Jelleo: the data layer that produces the signal an insurer or partner protocol uses to write coverage. Underwriting = deciding who/what to insure and at what price.
Cryptographically-signed attestation
A digital receipt cryptographically signed by Jelleo, proving "as of commit X, all critical invariants verified." Anyone can verify the signature without trusting Jelleo.
Code-level invariant integrity
Whether the rules the code is supposed to enforce (e.g., "vault total = sum of user balances") still hold. The thing Jelleo continuously checks and attests.
STRIDE evaluators
The assessors Solana Foundation uses to grade protocols under its STRIDE security program. They consume Jelleo attestations as one input to the smart-contract integrity pillar.
F7
Jelleo's first major bug discovery. F7 = a self-dealing insurance-siphon attack class in Percolator's use_insurance_buffer helper. Disclosed to Anatoly's team April 2026.
Residual-conservation insurance-siphon
F7's full label. "Residual conservation" was the broken invariant; "insurance siphon" is what the attacker did with it (drained the insurance buffer via self-trades).
Percolator
Anatoly Yakovenko's perpetual decentralized exchange (perp DEX) on Solana. Jelleo's first deployment target.
Anatoly Yakovenko
Co-founder of Solana. The audience for this proposal. Owner of Percolator (Jelleo's inaugural deployment).
SAFE
Simple Agreement for Future Equity. A startup funding instrument: investor wires money now, gets equity at the next priced round at a "valuation cap." YC-created, industry standard.
Pillars (P1–P4)
The 4 product capabilities: P1 counterfactual mainnet detection · P2 cross-protocol bug-class propagation · P3 closed-loop fix bundle · P4 on-chain attestation registry.
PR #39
The GitHub Pull Request where F7 was disclosed: aeyakovenko/percolator-prog#39. Closed without merge — the maintainer chose existing engine defenses + regression coverage instead of the proposed vault-debit patch.
Commit a1afd2e
The Git commit (40-char hash, first 7 shown) on Percolator's main branch where regression tests for F7 were formally labeled PR39/F7.

§01 Pages 3–4 — Executive Summary ~50 terms

Operational reference / data room
Collection of supporting docs, websites, and code that an investor can browse during diligence. Jelleo's data room: jelleo.com + GitHub repos + F7 disclosure PR.
Cluster status
Live page on jelleo.com showing the operational health of the audit pipeline (uptime, last cycle, queue depth). "Cluster" here = the connected set of agents and infrastructure.
Coverage map
Visual layout of which Percolator code paths Jelleo's hyp library currently covers. Lets a reader see "we look here, here, and here."
Disclosure ledger
Append-only log of every bug Jelleo has disclosed (currently: F7). Each entry timestamped + signed.
Attestation registry
The on-chain database (Pillar 4) of Jelleo's signed attestations. Other Solana programs can read it directly.
jelleo.com / jelleo.com/dashboard
The live operational reference site (cluster status, coverage map, disclosure ledger, attestation registry preview) and the customer dashboard demo (what a paying customer would see). Framed as funded-state Year-1 preview, not a public launch.
v0.1
Version 0.1 — early/MVP/prototype release of Jelleo, currently running.
125-hypothesis library
A catalog of 125 specific bug-pattern checks tailored to Percolator. Each hyp is a yes/no question about the code (does invariant X hold here?).
Hypothesis (hyp)
A falsifiable claim about code behavior — e.g., "the vault balance always equals the sum of user balances." Either holds or doesn't.
Cargo test
Rust language's built-in test runner. cargo test compiles and runs Rust tests. "Passing cargo test" = the test compiled and the assertion held.
Engine-class bug
A bug in the math/logic engine of a protocol (vs. a UI bug or a config bug). Engine bugs are the high-impact class because they affect every user.
Vault-debit patch
The specific fix Jelleo proposed for F7: when the insurance buffer absorbs loss, the vault must be debited too (so the residual counter and vault stay in sync). Maintainer chose regression coverage instead.
Regression coverage
Tests that fail if the bug ever comes back. Doesn't prevent the bug structurally — just catches re-introduction in CI.
Code-grounded
Anchored in actual source code — not assumptions, not models. Every claim cites file:line.
Counterparty
The other party in a transaction — in DeFi insurance, the entity that takes on risk in exchange for a premium.
Live signal
A real-time data feed that an external system can subscribe to and react to. Jelleo's attestations are a live signal.
Invariant
A property that should ALWAYS be true in the protocol — e.g., total supply = sum of balances. A bug = an invariant violation.
Conservation invariant
A subtype of invariant: something that should remain constant. F7 was a conservation-invariant bug — money disappeared from the insurance pool without the vault tracking it.
Residual conservation
The specific F7 invariant that broke: the protocol's internal "haircut residual" counter must equal the sum of all insurance-buffer absorptions. F7 showed it could grow unbounded — i.e., the protocol thought it had absorbed more loss than the vault actually paid out — letting an attacker claim the gap.
Haircut
In Percolator, the slice of a trade that gets routed to the insurance buffer (instead of paid out) when a position is socialized-loss-exposed. The "haircut residual" is the running tally of these slices.
Vault
The on-chain account holding actual user funds. In F7, the bug was that the residual counter advanced even when no funds left the vault — creating a "phantom" liability the attacker could redeem against.
Engine math
The arithmetic + state-transition logic at the core of a protocol. Where conservation invariants live and where F7-class bugs hide.
Protocol-state transitions
How the protocol moves from one state to another (e.g., trade executes, liquidation triggers). Bugs often live in transitions.
Attack vector
The pathway an attacker exploits — what input + what conditions cause harm.
Smart-contract integrity surface
The attack surface where contract code logic itself can be exploited — as opposed to "operational surface" (multisig theft, social engineering). STRIDE separates these explicitly.
Continuous code-grounded attestation
An ongoing cryptographic receipt proving "as of commit X, all critical invariants verified." Subscribable as a feed. The base unit Jelleo sells.
Underwriting primitive
The atomic unit insurers use to make underwriting decisions. Today on Solana: stale PDF audit. With Jelleo: live attestation.
Capital-constrained vs signal-constrained
"Capital-constrained" = not enough money. "Signal-constrained" = not enough information. Academic + industry consensus: DeFi insurance is mostly the former — Jelleo addresses the latter.
Reinsurance
Insurance for insurers. Traditional finance term. Big insurance pools come from reinsurance markets — a potential capital partner for crypto-native underwriters.
Two-sided market
A platform where two distinct customer types each pay/use it differently — here: protocols pay to be attested, insurers pay to consume the registry.
Nexus Mutual
The largest existing DeFi insurance protocol. Ethereum-native. Pioneered cover-mutual model.
InsurAce / Bridge Mutual
Existing multi-chain DeFi insurance protocols. Mostly oracle-parametric coverage.
Amulet / AmuShield
Solana-native DeFi insurance protocol. AmuShield is their coverage product. Closest existing peer to Jelleo's eventual customer set.
Oracle-parametric coverage
Insurance triggered by an oracle (price feed) — e.g., depeg of stablecoin triggers payout. Doesn't actually look at the code.
Audit-anchored coverage
Insurance where the underwriter relies on a one-time audit report. Stale on the next commit.
Invariant-violation parametric / Invariant-anchored coverage
Jelleo's target — insurance where the underwriter has a live signal of whether code-level invariants currently hold. Triggers payout on invariant breach detected.
Sherlock
Ethereum-native DeFi audit-contest + underwriting platform. Pioneered the model Jelleo extends to Solana.
OtterSec / Asymmetric Research / Zellic
Three of the top-tier Solana audit firms. Do $60K–$200K point-in-time engagements. Asymmetric also runs the STRIDE assessment program.
Neodyme / Halborn / Trail of Bits
More security firms. Neodyme = Solana-native (also runs Riverguard + SIRN). Halborn = multi-chain mid-tier. Trail of Bits = enterprise-tier ($200K+ engagements).
Point-in-time engagement
An audit done at a single moment — produces a PDF report that goes stale on the next code commit. Opposite of continuous.
FV incumbents (Certora / Runtime Verification)
Formal-verification firms that operate one tier deeper than Kani-style FV. Certora uses a CVL spec language; Runtime Verification uses the K Framework. High rigor, high cost.
Continuous-detection plays (Hypernative / Riverguard / Range Security / Sec3 X-Ray)
Other security products SF-funded under STRIDE. Hypernative + Range = runtime/perimeter monitoring (watch live transactions). Riverguard = Neodyme's attack-simulation product. Sec3 X-Ray = pre-deployment static analysis. All structurally complementary to Jelleo's commit-anchored invariant attestation, not competing.
Commit-anchored
Tied to a specific Git commit SHA. Jelleo attests "as of commit X." Different surface from runtime monitoring (which watches live txs without referencing source commits).
Runtime / perimeter / static-analysis surfaces
Three different security vantage points. Runtime = watching live behavior. Perimeter = watching boundaries (RPC, accounts). Static analysis = scanning code without running it. Jelleo lives in a fourth surface: continuous formal-style invariant checking on commit.
Pillar 3 fix bundles
A complete remediation package: the bug + AI-generated fix + Kani mathematical proof + tests, all opened as one Pull Request. Jelleo's actual remediation deliverable (vs. just a report).
Pillar 4 attestations
Cryptographic receipts published on Solana mainnet by Jelleo, queryable by any program. The consumable signal insurers and partner protocols ingest.
Solana mainnet
The live, production Solana blockchain (vs devnet which is for development). Real funds, real users.
STRIDE
Solana Trust, Resilience and Infrastructure for DeFi Enterprises. Solana Foundation security program launched April 6 2026, run by Asymmetric Research. 8 pillars including smart-contract integrity.
Tier 1 / Tier 2 (STRIDE)
STRIDE's customer tiers based on TVL: Tier 1 = $10M+ TVL (gets opsec). Tier 2 = $100M+ TVL (additionally gets formal-verification funding from Solana Foundation).
TVL
Total Value Locked — total dollar value of assets deposited in a DeFi protocol. The standard "size" metric.
Opsec
Operational security — protecting people, processes, infrastructure (vs code).
Formal verification (FV)
Mathematical proof that code correctly implements its specification. Stronger than testing — testing samples behavior; FV proves it across all inputs.
Program assessor
The body that grades vendors/protocols against a program standard. For STRIDE, Asymmetric Research is the assessor. Jelleo is a vendor producing artifacts the assessor evaluates.
Artifact-recognition channel
Year 1 OKR: open a working relationship with STRIDE assessors so Jelleo attestations are accepted as Tier 2 input. Outreach begins after registry hits mainnet (day 180).
Reimbursement path
Solana Foundation's Tier 2 FV grant could in principle be directed toward Jelleo's service on behalf of qualifying customers. No formal partnership today.
OKR
Objectives and Key Results. A goal-setting framework popularized by Google/Intel. Year 1 OKRs = the public goals Jelleo commits to in the 12-month engagement.
Post-money cap
SAFE valuation cap including the SAFE money itself. $1.32M / $10M post-money = ~13.2% stake at conversion.
Tranche
A chunk of money. Two-tranche SAFE = money split into two payments tied to milestones.
Tranche 2 trigger
The milestone (day 90 here) that releases the second half of the SAFE money. Hit P3 MVP + P4 devnet MVP = trigger fires. Miss = doesn't fire.
MVP
Minimum Viable Product — earliest version that demonstrates the value proposition. Used for both P3 and P4 milestones.
Devnet
Solana's development blockchain — like a sandbox, not real money. Distinct from mainnet (live, production).
Post-external-audit
After a third-party security firm has audited and signed off. P4 mainnet deployment is gated on this (day 180).
SLA
Service Level Agreement — contractual response-time guarantees. "First-call SLA" = Jelleo gets the first call when something breaks at the customer protocol.

§02 Page 5 — Methodology ~22 terms

Codified methodology
The discovery process that produced F7, written down + automated. Not service-hours — codified pipeline.
Ceiling-depth
The deepest tier of the audit pipeline (1,500+ hyps/day with multi-round refinement, attack-chain enumeration, and Kani synthesis on every Critical/High). Kicks in at M4 post-Tranche 2.
4-step pipeline
The discovery process: 1) invariant catalog → 2) tool-using verification → 3) empirical confirmation → 4) sibling derivation.
Invariant catalog
The list of code properties that should always hold. Each entry is a falsifiable claim with file/line targets and severity.
Hand-curated invariants
Hypotheses written by humans (vs. AI-generated). Higher signal density than auto-generated ones.
Surface family
A category of attack surfaces — e.g., conservation, oracle manipulation, governance. The 125 hyps span ~10 surface families.
Falsifiable claim
A statement that can be definitively proven true or false (Karl Popper's term). Opposite of vague prose.
File:line
A precise source-code reference like engine/src/lib.rs:4256. Required for every Jelleo claim — no speculation.
Severity
Bug-importance classification: Critical / High / Medium / Low / Info.
Tool-using verification
An AI agent uses tools (read_file, grep, find_function) to investigate code. Versus passive prompting where AI just outputs an answer.
Claude agent
An instance of Anthropic's Claude AI configured as an autonomous worker — runs a loop of "think → call tool → see result → think again."
read_file / grep / find_function
The three core tools the agent has: read a file's contents, search for a pattern, find a function by name.
Code paths
The execution sequences through source code. Walking a code path = following what happens step-by-step.
Line-cited verdict
The agent's final answer, with citations of the exact source-code lines that justify the conclusion.
Empirical confirmation
Proving a bug by actually running test code that triggers it — not just reasoning about the code.
Custom cargo test
A handwritten Rust test (using cargo test) that the AI authored to verify a specific bug claim.
Engine helpers
Internal utility functions in Percolator's engine code (e.g., use_insurance_buffer, resize_position).
PASS / FAIL classification
Test outcome. PASS = the bug claim was confirmed (test triggers the bug). FAIL = it didn't reproduce.
examples/confirmed-tests/
Public folder in the audit-pipeline-cli repo containing 3 Rust tests that AI generated and that pass against Percolator's real engine.
Sibling derivation
When a bug is confirmed, the AI generates 3–5 related hypotheses checking if the same pattern exists elsewhere in the code. Each goes back into the loop.
12-hyp sibling pack
The 12 follow-up hypotheses derived from F7 — checking similar conservation-violation patterns across other helpers.
Library compounds
The hyp library grows in value over time as confirmed findings derive sibling hyps that are higher signal-density than random ones.

§03 Page 6 — Learning Loop ~28 terms

Generation throughput
Speed of producing new hypotheses — measured at ~50 hyps/hour during F7 sibling derivation.
Steady-state cap (1,500/protocol)
The maximum library size at equilibrium — 1,500 hyps per protocol. Reached on disk by month 1.
Library on disk vs daily invocation rate
Two different things: how many hyps are stored locally (1,500) vs how many we actually run per day (500 → 1,500). Library = generation cost (one-time). Invocations = compute bill (per-day).
Production tier (500/day)
Compute level: running 500 hyps per day. Used during M1–3 build phase for sibling-validation cycles.
Ceiling tier (1,500+/day)
Compute level: running 1,500+ hyps per day with multi-round refinement, attack-chain enumeration, and Kani synthesis on every Critical/High finding.
Multi-round refinement
Iterating multiple times on a hypothesis to get a sharper verdict (vs single-shot). Costs more, signal is higher.
Attack-chain enumeration
Systematically listing all the multi-step exploit sequences that could lead to a bug (rather than treating each hyp in isolation).
Kani synthesis
Auto-generating Kani proof harnesses from confirmed findings. Kani then mathematically verifies the harness.
Critical / High
The top two severity levels. Only these get the full Kani synthesis treatment in Ceiling tier.
Cluster catalog
The combined hyp libraries across all protocols Jelleo covers. Scales linearly with each protocol added.
Multi-protocol coverage
Year 2+ scope: cover multiple protocols, not just Percolator. Cluster catalog reaches 3,000–8,000 hyps total at that point.
Cross-protocol propagation
When a bug is found in one protocol, search for the same pattern in all other indexed protocols. Amortizes generation cost across the cluster.
Rolling catalog
The library is a rolling catalog (entries replace each other as quality improves), not a unidirectional accumulator (only growing).
Sibling-confirmation rate
How often AI-derived sibling hyps actually find a real bug. Conservative model assumes 10–15%.
Empirical noise-floor data
Measured baseline of false positives — collected during the funded engagement to calibrate library quality.
Attack class
A category of bug type. Five classes today: state-transition, invariant-property, implicit-invariant, arithmetic-overflow, authorization.
State-transition
A bug where the protocol state moves from one valid configuration to an invalid one (a transition that shouldn't be reachable).
Invariant-property
A bug where a stated property (e.g., total = sum of parts) is violated.
Implicit-invariant
A bug where a property the code assumes is true (without explicitly checking) is violated.
Arithmetic-overflow
Math result exceeds the variable's size, wrapping around or panicking. Classic bug class in Rust/Solana.
Authorization
A bug where the code doesn't properly check who can do what (e.g., signer not verified before mutation).
Sub-families
Sub-categories of the 5 main attack classes — ~30 total as disclosures derive new ones.
Conservation violation
Bug where something that should remain constant doesn't. F7's class.
Cursor-wrap state desync
A bug where a counter wraps around and state goes out of sync.
Oracle manipulation
Exploit of price-feed sources — e.g., flash-loan to move a price, then drain protocol assuming wrong price.
Liquidation race
Timing bug around forced position closures — multiple actors race to claim, one ends up with bad outcome.
Signature replay
Reusing a previously-valid signature to authorize an unintended action.
Governance hijack
Taking over protocol governance — e.g., flash-loan-borrowed tokens vote on a malicious proposal.

§04 Page 7 — Architecture · Four Pillars ~22 terms

Interlocking pillars
The 4 pillars feed each other: P1 detection → P2 propagation → P3 fix bundle → P4 attestation. Output of one is input to the next.
P1 · Counterfactual mainnet detection
For every mainnet transaction, simulate the alternate scenario in parallel. If state diverges from expected, flag the transaction in real time — within seconds of slot finality.
Per-tx parallel simulation
Running a simulation for each transaction alongside actual mainnet. "Per-tx" = one simulation per transaction. "Parallel" = simultaneously, not after the fact.
State divergence
When actual on-chain state differs from the expected/simulated state. The signal that something is wrong.
Slot finality
The moment a Solana slot (block) is confirmed final. Jelleo flags within seconds of this.
Sub-slot replay
Funded delta on P1: replay transactions faster than one slot (~400ms) so detection is in real time, not after.
shadow command
Existing CLI primitive in audit-pipeline-cli. shadow is the foundation P1 builds on (commit-triggered today; mainnet-ingestion in funded version).
P2 · Cross-protocol bug-class propagation
When a bug is disclosed anywhere (Immunefi, public PR, advisory), Jelleo extracts the structural pattern and searches every indexed protocol for the same class.
Immunefi
The major bug-bounty platform for crypto. Source of public bug disclosures Jelleo mines for new patterns.
Structural attack pattern
The abstract shape of an exploit — e.g., "trust an unverified oracle in math operation X" — independent of the specific protocol.
Indexed protocol
A protocol that Jelleo has cataloged in its index — meaning Jelleo can search its code structurally.
propagate / learn-from-disclosures
Two existing CLI commands. propagate = run a known pattern across all indexed protocols. learn-from-disclosures = mine public bug reports for new patterns.
Disclosure-feed integration
Funded delta on P2: live ingestion from Immunefi + GitHub advisory + public PR feeds (vs. manual feed reads).
P3 · Closed-loop fix bundle
Bug → AI generates a candidate fix → Kani harness mathematically proves the fix preserves every other invariant → tests run → bundle (bug + fix + proof + tests) opens as a single PR.
Candidate fix
The proposed code change. AI-generated, but human-reviewable.
Kani harness
A test setup written for the Kani verifier. The harness defines what's input, what's output, and what should be true.
Kani / Kani Verifier
A model checker for Rust developed by AWS. Mathematically explores all input combinations to prove (or disprove) properties.
Single PR
One Pull Request packaging everything together — easier for the maintainer to review and merge atomically.
confirm / synth-kani
Two existing CLI commands. confirm = autonomous test-writing loop. synth-kani = NL → Kani harness synthesis with compile-fix-retry.
P4 · On-chain attestation registry
Every audit cycle publishes a cryptographically-signed Merkle root attesting "these invariants verified at this commit SHA." On Solana mainnet. Queryable by any program.
Merkle root
A cryptographic summary of many records — a single hash that proves all the records are intact. Standard blockchain primitive.
Commit SHA
Git's unique 40-character identifier for a specific code version. Jelleo attests "as of this SHA, the invariants held."
Composable
Can be used as a building block by other software. P4 attestations are composable because any other Solana program can read them and react.
Anchor program / Anchor framework
The standard Solana smart-contract framework. Higher-level than raw Rust + Solana SDK. Most modern Solana programs use Anchor. P4's ~500-LOC attestation registry is an Anchor program.
Ed25519
Modern public-key signature algorithm. Widely used in crypto/Solana. Jelleo signs every disclosure with an Ed25519 key (off-chain in v0.1; on-chain in funded P4).

§05 Page 8 — Today vs Funded ~18 terms

Engine + wrapper
Percolator's two parts: the off-chain engine (does the math) + the on-chain wrapper (Solana program that interfaces with it). Jelleo's 12-month scope covers both.
Auto-PoC
Automatically-generated Proof of Concept — a working test that demonstrates a bug. PoC = "proof this isn't theoretical."
cargo PASS
The Rust test runner reported PASS — the test compiled, ran, and confirmed the bug claim.
6 cycles · $200 spend
6 audit runs costing $200 total in Anthropic API. Demonstrates the unit economics work at small scale.
Sonnet 4.6
Anthropic's mid-tier Claude model. Cheaper, fast, used for most hyp runs.
Opus 4.7
Anthropic's top-tier Claude model. ~3× more expensive than Sonnet, used on confirmation and Kani synthesis.
Single-shot
One pass — no iteration. Cheaper but lower-quality output than multi-round.
Confirm loop with cargo-feedback
Autonomous loop: AI writes a test → compiles → if compile error, AI reads the error and fixes → repeats. Reaches ~95% PASS rate.
LiteSVM
Embedded Solana VM library. Lets tests simulate the full Solana runtime (BPF) inside a Rust process — much faster than spinning up a real validator.
BPF
Berkeley Packet Filter — the bytecode format Solana programs compile to. LiteSVM simulates the BPF runtime so reachability is verifiable end-to-end.
LiteSVM exploit chains
Multi-step attack reproductions using LiteSVM — proves a real exploit works, not just that math is wrong.
Hunt cycle
A complete Jelleo audit pass: recon → debate → PoC → Kani → DB → alert.
Commit-triggered
A hunt cycle starts automatically every time the target repo gets a new commit.
Kani-verified corpus
The hypothesis library with Kani formal proofs attached to each entry — strongest possible verification.
Slack webhook
Automated message posting to Slack via a webhook URL. The standard way to send real-time alerts.
GitHub auto-issue
Automatically filing a GitHub Issue against the target repo when a bug is confirmed. Lifecycle transitions to "disclosed."
SQLite DB
Lightweight embedded SQL database. Used in v0.1 for storing findings.
Postgres
Production-grade SQL database. Funded version migrates from SQLite to Postgres for scale.
Audit trail
Complete log of every state change in the findings DB — who/what/when. Required for trust.

§06 Pages 9–10 — Unit Economics ~28 terms

Per-MTok pricing
Per million tokens. Anthropic charges $3 input / $15 output per MTok for Sonnet 4.6, $5/$25 for Opus 4.7.
MTok
Million tokens. A token ≈ 0.75 words on average. Code is denser (more tokens per character).
Effective rate (2.25× Sonnet)
The actual cost factoring in tokenizer overhead, retry behavior, etc. Opus 4.7 sits at 2.25× Sonnet because of Rust tokenizer overhead.
Tokenizer overhead
Code (Rust) tokenizes into more tokens per character than prose, so technical content costs more than text. ~35% overhead for Rust.
Speculation recon
Cheap initial pass: ~$0.05/hyp Sonnet. Quick scan to filter out obviously-wrong hypotheses.
Refinement round
A secondary thinking pass that improves verdict quality. Adds modest cost ($0.08/hyp on Sonnet).
Tool-using deep audit (15–25 turns)
Extensive code reading via tools (15–25 turns of read_file/grep). The most expensive recon mode but highest signal: ~$1.50/hyp Sonnet.
Custom PoC + cargo run
Writing and running a real Rust test. ~$2/hyp Sonnet, ~$4.50/hyp Opus.
Exploit-chain authoring
Writing a multi-step LiteSVM attack reproduction. Most expensive mode at ~$3/hyp Sonnet, ~$6.75/hyp Opus.
Worst-case headroom
Extra budget allocated for the worst-case scenario. $100K above the $525K base = $625K total in the API line.
Prompt caching (0.1× base input)
Anthropic feature: when you reuse the same long context, input cost drops to 10% of normal. Massive win for Jelleo's "same engine code, many hyps" pattern.
Cache miss / write-tier surcharge (1.25× base input)
The other side of prompt caching: the FIRST time a context is written to the cache, you pay 1.25× the base input rate (a "write fee" Anthropic charges to provision the cache slot). Subsequent reads from that slot are 0.1× until the slot expires. Each new protocol pays a one-time premium to warm the engine-context cache, then runs cheap until invalidation.
Batch API (50% off)
Anthropic feature: non-urgent requests batched together cost 50% of normal. Used for non-real-time hyp runs.
Reusable-context pattern
Same engine source code is loaded once, used across many hypotheses. Pairs perfectly with prompt caching.
Cache-hit rate (~70%)
The fraction of input that hits the cache after warmup (4-week target). Drives effective per-hyp cost downward.
Per-hyp blended cost
Average cost per hyp under the assumed mix of Sonnet/Opus + cache + batch. Foundation ~$1.39/hyp, Production ~$1.67/hyp, Ceiling ~$1.10–$1.55/hyp.
Customer pricing tiers
Foundation / Production / Ceiling — the same depth tiers Jelleo runs internally also serve as customer-facing pricing options.
Hetzner AX102
Hetzner's high-end dedicated server class. AMD Ryzen 9 7950X3D-class machine — runs the hunt loop, Kani dispatch, LiteSVM reproduction, and dashboard.
Dedicated server
A physical machine rented from a hosting provider (vs. shared VM/cloud instance). Predictable performance, fixed cost.
Hetzner Storage Box
Hetzner's backup storage product — separate-DC location, used for nightly automated Postgres + filesystem backups.
Separate-DC backups
Backups stored in a different data center from the primary. Survives single-DC failure.
RPO ≤24h
Recovery Point Objective ≤ 24 hours — if disaster strikes, we lose at most 24h of data.
Restore-tested
Backups are periodically restored to verify they actually work. Untested backups are not real backups.
Observability
Tooling to see what's happening inside the system in real time — metrics, logs, traces. Implemented via Grafana Cloud + Slack + Cloudflare.
Grafana Cloud
Hosted version of Grafana — monitoring dashboards. Used for observability of the cluster.
Cloudflare
CDN + DNS + DDoS protection provider. Sits in front of jelleo.com.
SLA-grade hot-standby
A second data center with a live copy of the primary, ready to take over within seconds. ~$300/mo upgrade activated when first paying customer requires SLA.
COGS
Cost of Goods Sold — the direct cost to deliver one unit of service. For Jelleo: API + infra. The basis for §07's 2.4× cost-plus pricing.

§07 Pages 11–13 — Revenue Model ~30 terms

Continuous coverage agreement
Ongoing monthly contract — protocols pay every month for continuous Jelleo coverage. Not a one-time audit.
Cost-plus per engagement
Pricing model: customer pays our raw cost + a fixed markup. Transparent, defensible, bounded.
Quote-based pricing
Pricing set per engagement (vs. fixed list price). What audit firms do today; Jelleo follows the same model with the cost-plus formula behind it.
2.4× revenue markup

Customers can move between tiers month-over-month. Engagement length is open-ended.

Average cost across different operations (Sonnet vs Opus, single-shot vs deep audit, cache-hit vs cache-miss). The blended figure is what drives per-hyp economics.

What $1B+ TVL Solana protocols spend $150K–$500K/yr on today: separate audit firms + ongoing retainer relationships + Immunefi bug-bounty programs. Jelleo Production at $720K/yr replaces this with continuous depth.

Open-source AI models (Llama, Mistral, etc.) where the model file is publicly available. Fallback if closed providers become unviable.

Risk of being permanently dependent on one provider. Hedged at the abstraction layer (code can switch) not at the model (we use Anthropic today).
